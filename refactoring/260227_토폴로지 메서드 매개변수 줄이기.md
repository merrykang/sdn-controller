# TreeTopologyService 매개변수 정리

## 1. 호출 관계 요약

`createSwitchAndHostTopology(School school)` 한 곳에서만 사용되는 private 메서드들:
- getDepthMap, detectSwitchAndHostIpConflicts, addHostNodes, addHubNodes, addSwitchNodes
- buildMgmtIpIndices, collectAllLinks, removeDuplicateLinks, addEdgesFromLinks
- getSwitchIps, getVlanIds, getVnNames, getPortStats, propagateVlanTo* 등

위 메서드들이 서로 공유하는 데이터가 많아서 **같은 빌드 컨텍스트**로 묶을 수 있음.

---

## 2. 줄일 수 있는 매개변수별 판단

| 데이터 | 현재 전달되는 메서드 | 인스턴스 변수 추천 | 판단 이유 |
|--------|----------------------|--------------------|-----------|
| **switchList** | getDepthMap, detectSwitchAndHostIpConflicts, addHostNodes, addHubNodes, addSwitchNodes, collectAllLinks, addEdgesFromLinks, buildMgmtIpIndices, getSwitchIps | ✅ **인스턴스** | 이미 `topologyDeviceData`가 인스턴스에 있음. `getSwitchList()`로 매번 꺼내 쓰거나, 빌드 시점에 한 번만 `List<SwConfTotalSwitch> switchList`를 인스턴스에 두고 사용 가능. **단일 학교 토폴로지 빌드** 동안만 유효하므로 인스턴스로 두어도 스레드 안전성은 createSwitchAndHostTopology 호출 단위로 유지됨. |
| **hostList** | getDepthMap, addHostNodes, collectAllLinks, addEdgesFromLinks, propagateVlanToHubs | ✅ **인스턴스** | switchList와 동일. `topologyDeviceData.getHostList()` 또는 빌드용 인스턴스 필드로 통일. |
| **hubList** | getDepthMap, addHubNodes, collectAllLinks | ✅ **인스턴스** | 위와 동일. |
| **schoolMark** | getDepthMap, addSwitchNodes(school에서 추출), addEdgesFromLinks(school에서 추출), swagService/swagLinkService 호출 등 | ✅ **인스턴스** | 한 번의 `createSwitchAndHostTopology` 호출당 하나의 학교. `currentSchoolMark` 또는 `currentSchool`만 두고 나머지는 여기서 꺼내 쓰면 됨. |
| **school** | addSwitchNodes, addEdgesFromLinks (getSchoolMark(), getEduOffice() 등) | ✅ **인스턴스** | schoolMark만 쓴다면 `currentSchoolMark`만 있어도 되고, getEduOffice() 등이 필요하면 `currentSchool`을 인스턴스로 두는 편이 자연스러움. |
| **hostCnt** | addHostNodes, addHubNodes, collectAllLinks | ✅ **인스턴스** | 한 번 계산 후 여러 메서드에서만 읽음. 빌드 컨텍스트용 인스턴스 변수로 적합. |
| **conflictedIps** | addSwitchNodes, addHostNodes, addHubNodes | ✅ **인스턴스** | 한 번 계산 후 읽기만 함. 인스턴스로 두면 매개변수 3곳에서 제거 가능. |
| **depthMap** | addEdgesFromLinks | ⚠️ **둘 다 가능** | 사용처가 addEdgesFromLinks 하나. 인스턴스로 두면 “현재 빌드의 depthMap”이 명확해지고, 매개변수로 두면 의존성이 메서드 시그니처에 드러남. **권장: 인스턴스** (다른 빌드 데이터와 일관성). |
| **switchLinkMap, hostLinkMap** | addEdgesFromLinks | ✅ **인스턴스** | 한 번 조회 후 addEdgesFromLinks에서만 사용. 빌드용 인스턴스 변수로 두면 addEdgesFromLinks 시그니처가 짧아짐. |
| **linkSet / noDuplicateLinkSet** | addEdgesFromLinks | ✅ **인스턴스** | 위와 동일. “현재 빌드의 링크 집합”을 인스턴스로 두는 것이 자연스러움. |
| **nodes, edges** | addHostNodes, addHubNodes, addSwitchNodes, addEdgesFromLinks | ❌ **매개변수 유지** | 빌드 결과를 담는 **가변 컬렉션**. 인자로 넘기는 편이 “이 메서드가 이 리스트에 추가한다”는 의도가 명확하고, 테스트·재사용 시에도 유리. 인스턴스로 두면 생성 직후 초기화/초기화 해제를 놓치기 쉬움. |

## 3. 정리 (권장 방향)

### 인스턴스 변수로 두면 좋은 것 (빌드 컨텍스트)

- **이미 있음:** `topologyDeviceData`
- **추가 권장 (한 학교 토폴로지 빌드 동안만 사용):**
  - `currentSchool` 또는 `currentSchoolMark` (그리고 필요 시 `currentSchool`)
  - `switchList`, `hostList`, `hubList` — 또는 `topologyDeviceData`만 두고 `getSwitchList()` 등으로 접근
  - `hostCnt`
  - `conflictedIps`
  - `depthMap`
  - `noDuplicateLinkSet` (또는 linkSet)
  - `switchLinkMap`, `hostLinkMap`

이렇게 하면:

- `getDepthMap(schoolMark, switchList, hostList, hubList)` → `getDepthMap()` (또는 `getDepthMap(schoolMark)` 만)
- `detectSwitchAndHostIpConflicts(switchList, hostList)` → `detectSwitchAndHostIpConflicts()`
- `addSwitchNodes(nodes, edges, school, switchList, conflictedIps)` → `addSwitchNodes(nodes, edges)`
- `addHostNodes(nodes, edges, switchList, hostList, hostCnt, conflictedIps)` → `addHostNodes(nodes, edges)`
- `addHubNodes(nodes, edges, switchList, hubList, hostCnt, conflictedIps)` → `addHubNodes(nodes, edges)`
- `collectAllLinks(switchList, hostList, hostCnt, hubList)` → `collectAllLinks()`
- `addEdgesFromLinks(school, switchList, hostList, edges, noDuplicateLinkSet, depthMap, switchLinkMap, hostLinkMap)` → `addEdgesFromLinks(edges)`
- `buildMgmtIpIndices(switchList)` → `buildMgmtIpIndices()`
- `getSwitchIps(switchList)` → `getSwitchIps()` (내부에서 인스턴스 switchList 사용)

등으로 시그니처를 크게 줄일 수 있음.

### 매개변수로 유지하는 것이 좋은 것

- **nodes, edges**  
  - “어디에 노드/엣지를 채울지”를 명시적으로 넘기는 편이 유지보수와 테스트에 유리.

## 4. 주의사항

- **인스턴스 변수 사용 구간:** 위 인스턴스 변수들은 **`createSwitchAndHostTopology` 진입 시 설정하고, 해당 메서드와 그 안에서만 호출되는 private 메서드에서만 사용**하는 것이 안전함. 다른 public 메서드에서 이 필드를 재사용하면 동시성·재진입 이슈가 생길 수 있으므로, “한 학교 빌드” 스코프로만 쓰는 것이 좋음.
- **topologyDeviceData:** 이미 인스턴스에 있으므로, `switchList`/`hostList`/`hubList`를 별도 인스턴스에 둘지, 아니면 `topologyDeviceData.getSwitchList()` 등으로만 접근할지는 팀 컨벤션에 맞춰 선택하면 됨. (가독성 측면에서는 `getSwitchList()` 한 번만 호출해 지역변수/인스턴스에 담아 두는 편이 반복 호출을 줄일 수 있음.)

이렇게 적용하면 매개변수 개수를 많이 줄이면서도, “한 번에 한 학교 토폴로지”라는 스코프가 명확해짐.
